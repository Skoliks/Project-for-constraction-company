# Code Review

## Краткая оценка

Проект уже решает полный рабочий сценарий: парсер собирает данные, сервисный слой обрабатывает их, данные сохраняются в БД и доступны через FastAPI API. Для pet-project это сильная база: доменная модель выделена, CRUD вынесен по слоям, парсер поддерживает пагинацию, ограничение параллелизма и базовую дедупликацию.

Главная проблема текущей реализации не в отсутствии структуры, а в надежности на границах системы: транзакционность импорта, поведение при частичных ошибках, способ запуска долгого парсинга из HTTP-запроса и слабая подготовка к расширению на нескольких поставщиков.

## Сильные стороны проекта

- Архитектура уже разложена по слоям: `routes -> services -> repositories -> models`, и это упрощает навигацию по коду.
- Для предметной области выделены отдельные сущности `suppliers`, `materials`, `supplier_products`, `material_prices`, а не одна "универсальная" таблица.
- В сервисах есть базовая бизнес-валидация перед записью в БД: проверка существования поставщика, материала и конфликтов по уникальным полям.
- В `app/parsers/stroylandia_parser.py` есть ограничение конкурентности через `asyncio.Semaphore` и `aiohttp.TCPConnector(limit=...)`, что лучше, чем безлимитный `gather`.
- Пагинация сделана не только через ссылки из HTML, но и через fallback с `PAGEN_1`, что повышает устойчивость к неполному DOM.
- Импорт в `app/services/stroilandia_import_services.py` считает метрики результата: сколько записей создано, переиспользовано и пропущено. Для мониторинга ETL это полезно.
- Схемы Pydantic отделены от ORM-моделей, поэтому API-контракты не смешаны с таблицами.
- Есть health-check для приложения и отдельная проверка подключения к БД.

## Слабые стороны

- Репозитории сами вызывают `commit()`. Из-за этого сервисы теряют контроль над транзакцией и не могут атомарно сохранить связанные сущности.
- В проекте нет миграций. Таблицы создаются через `Base.metadata.create_all(bind=engine)` при старте приложения, что удобно только на раннем этапе, но плохо управляется в реальной эксплуатации.
- Логирование не внедрено: `app/core/logger.py` пустой, а парсер и seed-скрипт используют `print`.
- Нет тестов на сервисы, репозитории и парсер. При изменении HTML-структуры сайта регрессии будут ловиться только в runtime.
- CRUD-роуты возвращают полные списки без пагинации и фильтрации, что быстро станет узким местом при росте данных.
- Парсер Стройландии жёстко связан с одним поставщиком и прямым совпадением `material_name` с записью в таблице `materials`.
- Названия сервисов и файлов местами неравномерны: `materials_services.py`, `supplier_services.py`, `stroilandia_import_services.py`. Это не ломает код, но ухудшает единообразие.

## Возможные баги и риски

### 1. Обновление материала работает некорректно при patch без `name`

В `app/services/materials_services.py:35-52` сохранение изменений происходит только внутри блока `if update_data.name is not None`.  
Следствие:

- если клиент хочет обновить только `category`, `base_unit` или `description`, метод ничего не возвращает;
- роут `PATCH /materials/{id}` с `response_model=Material` с высокой вероятностью завершится `500 Internal Server Error`.

Проблемный фрагмент:

```python
if update_data.name is not None:
    ...
    return material_repository.save(db=db, material=material)
```

### 2. Импорт не атомарен и может оставлять "висячие" продукты без цены

В `app/services/stroilandia_import_services.py:50-84` сначала создается `supplier_product`, а затем отдельно создается `material_price`.  
Обе операции коммитятся внутри репозиториев (`app/repositories/supplier_product_repository.py:17-24`, `app/repositories/material_price_repository.py:14-21`).

Следствие:

- если продукт создан;
- потом `price_rub` оказался невалидным или следующая операция упала;
- в БД останется `supplier_product` без записи в `material_prices`.

Для ETL-сценария это риск нарушения целостности и непредсказуемых повторных импортов.

### 3. Долгий парсинг запускается прямо из HTTP-запроса

`app/routes/parsers.py:10-18` ожидает завершения всего импорта в рамках одного запроса.  
Следствие:

- запрос может жить минуты;
- воркер API занят долгой задачей;
- при таймауте reverse proxy пользователь увидит ошибку, хотя импорт частично уже мог записать данные в БД;
- повторный вызов легко создаёт гонки по тем же URL.

### 4. Привязка импорта к точному совпадению имени материала

В `app/services/stroilandia_import_services.py:30-38` материал ищется по `material_name`, пришедшему из парсера.  
Следствие:

- любое различие в названии между seed-данными и названием категории парсера приводит к `skipped_items += 1`;
- масштабирование на несколько поставщиков станет хрупким, потому что matching уже сейчас основан на строковом равенстве.

### 5. Дедупликация в парсере неполная

В `app/parsers/stroylandia_parser.py:243-305` дедупликация строится только на `(material_name, sku)`.  
Следствие:

- товары без `sku` не дедуплицируются вовсе;
- один и тот же товар с разными SKU или без SKU может попасть повторно;
- уникальность фактически переносится на `product_url`, но это происходит позже, уже в сервисе и через отдельный запрос в БД.

### 6. Неподготовленный второй парсер

`app/parsers/petrovich_parser.py` выглядит как исследовательский модуль, а не production-ready компонент:

- используется `headless=False` (`app/parsers/petrovich_parser.py:37-40`);
- модуль импортирует `playwright`, но этой зависимости нет в `requirements.txt`;
- есть неиспользуемый `requests`;
- `attributes_json` формируется как `dict`, тогда как схема ожидает `str | None` (`app/schemas/supplier_product_schema.py:5-12`).

Это пока не ломает основной поток, потому что модуль не интегрирован в API, но при подключении второго поставщика ошибка проявится сразу.

### 7. Создание схемы БД на старте приложения

`app/main.py:11` вызывает `Base.metadata.create_all(bind=engine)`.  
Риск:

- изменения структуры таблиц не отслеживаются как миграции;
- прод-окружение трудно воспроизводить;
- схема зависит от момента первого старта сервиса, а не от управляемого deployment step.

## Архитектурные замечания

- Слои выделены правильно, но границы ответственности пока не доведены до конца.
- Репозиторий должен отвечать за доступ к данным, а не за завершение транзакции. Сейчас транзакционный контроль размазан по всем `create/save/delete`.
- Импорт Стройландии лучше оформить как отдельный application service / job, а API-роут оставить тонким триггером.
- Для ETL-операций полезно отделить:
  - этап извлечения данных;
  - этап нормализации;
  - этап upsert в БД.
- Сейчас `supplier_products` хранит "сырой" товар поставщика, а `materials` играет роль нормализованного справочника. Это правильная идея, но ей не хватает явного слоя matching / mapping.
- `attributes_json` лучше хранить как `JSON/JSONB`, а не `Text`, иначе пропадает возможность нормального поиска и структурной валидации.

## Работа парсера

### Что сделано хорошо

- Асинхронный `aiohttp` вместо последовательного `requests`.
- Есть ограничение параллелизма.
- Есть fallback-логика по пагинации.
- Ошибки отдельных страниц не валят весь сбор: `asyncio.gather(..., return_exceptions=True)`.

### Что требует усиления

- Нет retry/backoff для временных сетевых ошибок и 429/5xx.
- Нет явного `timeout` на запрос уровня `session.get(..., timeout=...)`; используется только общий session timeout.
- Нет структурированного контекста ошибки: категория, страница, supplier, elapsed time.
- Нет отдельного этапа валидации извлечённых полей перед передачей в сервис.
- Парсер возвращает `list[dict]`, а не typed DTO/схемы, поэтому ошибки типов ловятся поздно.

## Работа с базой данных

### Плюсы

- Есть внешние ключи между таблицами.
- На `suppliers.name`, `suppliers.website_url`, `supplier_products.product_url`, `materials.name` заданы уникальные ограничения.

### Замечания

- Нет уникального ограничения на историю цен, например по `(supplier_product_id, observed_at)` или по политике "одна запись на запуск импорта".
- Нет индексов под ожидаемые запросы чтения, кроме индексов на `id`.
- Нет `cascade`/`passive_deletes` в связях, поэтому удаление поставщика или материала потенциально приведет к ошибкам FK или сиротским данным в зависимости от схемы БД.
- Нет rollback-обработки при `IntegrityError`.
- `SessionLocal = sessionmaker(bind=engine)` без явных `autoflush/autocommit/expire_on_commit` не критично, но поведение лучше задать явно.

## Читаемость и нейминг

- В целом код читается быстро: функции короткие, слои изолированы, названия сущностей предметные.
- Плюс: нейминг таблиц и маршрутов отражает бизнес-смысл.
- Минус: единообразие файлов страдает из-за смешения `service/services`, единственного `stroilandia_import_services.py` и множественного числа в названиях сервисов.
- Минус: сообщения об ошибках местами содержат опечатки: `alredy exist`.
- Минус: в части модулей много "сырого" procedural-style кода с `dict`, где уже просится отдельный DTO.

## Поддерживаемость кода

- Поддерживаемость выше среднего для pet-project за счет слоистой структуры.
- Основной фактор риска поддержки не в размере кода, а в отсутствии автоматических тестов и формализованных контрактов между parser/service/repository.
- При изменении HTML сайта придется руками искать, какой ключ в словаре перестал приходить.
- Отсутствие централизованного логирования усложнит диагностику сбоев импорта на сервере.

## Идеи по улучшению производительности

- Убрать `commit()` из каждого repository метода и перейти на батчевый commit на уровне сервиса импорта.
- Для импорта использовать bulk-стратегию: заранее загружать существующие `materials` и `supplier_products` в память словарями, а не делать запрос в БД на каждый item.
- Вынести поиск существующих URL в одну пачку `IN (...)`, а не выполнять `get_by_product_url()` по одному товару.
- Добавить batch insert для `material_prices`, если не требуется немедленный `refresh()`.
- Для больших объёмов API-списков добавить пагинацию и лимиты.
- При росте числа страниц стоит собирать категории конкурентно, но с отдельным rate-limit на домен.

## Замечания по масштабируемости

- Текущая модель хорошо подходит для одного поставщика, но при двух и более поставщиках начнет не хватать нормализации и явного matching layer.
- Прямой поиск `material` по имени не масштабируется: нужен словарь синонимов, rules-based matching или отдельная таблица сопоставлений.
- История цен уже выделена отдельно, и это хороший фундамент для аналитики, но без индексов и политики retention таблица быстро разрастется.
- HTTP-trigger для тяжёлого импорта не масштабируется по воркерам. Лучше переходить к background jobs / scheduler.
- Добавление новых поставщиков сейчас потребует копирования логики импорта вместо введения общего интерфейса `supplier parser -> normalized DTO -> import pipeline`.

## Suggested Next Steps

- Добавить второго поставщика через единый интерфейс импортера: общий контракт на выходе парсера, отдельная стратегия под каждый сайт.
- Усилить дедупликацию: использовать не только `sku`, но и `product_url`, нормализованное имя, а также защиту на уровне БД и upsert-политику.
- Перейти с `print` на нормальное логирование: уровни `info/warning/error`, request id, supplier, category, page_url.
- Ввести кэширование и/или планировщик: cron/APScheduler/Celery beat для регулярного обновления без ручного POST-запуска.
- Добавить парсинг карточек товаров: характеристики, бренд, упаковка, базовая единица, наличие, возможно изображение.
- Вынести нормализацию материалов в отдельный слой matching: словарь алиасов, правила сопоставления, ручное подтверждение спорных кейсов.
