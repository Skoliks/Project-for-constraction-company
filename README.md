# FastAPI Backend для мониторинга цен на стройматериалы

Backend-проект на FastAPI для сбора, нормализации и хранения цен на строительные материалы от внешних поставщиков.  
Текущий pipeline уже работает end-to-end:

`parser -> service -> PostgreSQL -> API`

Проект ориентирован на задачу агрегации товарных позиций поставщиков, хранения истории цен и подготовки базы для дальнейшего сравнения материалов между разными источниками.

## Описание проекта

Сервис собирает товары с сайта Стройландия, сопоставляет их с внутренним справочником материалов, сохраняет позиции поставщика в `supplier_products`, а историю наблюдаемых цен в `material_prices`.

Это не просто CRUD-приложение, а backend с реальным ETL-сценарием:

- есть асинхронный парсер;
- есть нормализованная модель данных;
- есть сервисный слой с бизнес-логикой;
- есть API для запуска импорта и доступа к данным.

## Основной функционал

- FastAPI API для работы с поставщиками, материалами, товарами поставщиков и ценами.
- Асинхронный парсер Стройландии на `aiohttp + BeautifulSoup`.
- Поддержка пагинации и ограничение числа одновременных запросов.
- Базовая дедупликация товаров при сборе.
- Сохранение истории цен в отдельную таблицу.
- Проверка существования связанных сущностей перед записью в БД.
- Health-check для API и подключения к PostgreSQL.

## Архитектура проекта

Проект разделён по слоям:

- `app/routes`  
  HTTP-роуты FastAPI.

- `app/services`  
  Бизнес-логика, валидация, orchestration между parser и repository.

- `app/repositories`  
  Работа с SQLAlchemy ORM и таблицами.

- `app/models`  
  ORM-модели БД.

- `app/schemas`  
  Pydantic-схемы для входных и выходных DTO.

- `app/parsers`  
  Логика парсинга внешних сайтов.

- `app/core`  
  Конфиг, подключение к БД, инфраструктурный код.

- `app/scripts`  
  Вспомогательные скрипты, например seed стартовых данных.

### Поток данных

1. Клиент вызывает API-роут запуска парсера.
2. Сервис импорта запускает асинхронный сбор страниц.
3. Парсер возвращает нормализованный список товаров.
4. Сервис ищет нужный `supplier` и `material`.
5. Товар поставщика сохраняется в `supplier_products`.
6. Цена сохраняется в `material_prices`.
7. API возвращает итоговую статистику запуска.

## Как работает парсер

Текущая рабочая интеграция реализована для Стройландии.

Что делает парсер:

- проходит по заранее заданным категориям стройматериалов;
- загружает первую страницу категории;
- определяет общее число страниц;
- формирует список pagination URL;
- асинхронно забирает остальные страницы;
- извлекает:
  - название товара;
  - URL товара;
  - SKU;
  - цену;
  - старую цену;
  - единицу измерения;
- выполняет базовую дедупликацию по SKU внутри категории;
- передаёт результат в сервис импорта.

### Почему данные разделены на две таблицы

- `supplier_products` хранит саму товарную позицию поставщика.
- `material_prices` хранит историю наблюдений цены во времени.

Это позволяет:

- не дублировать описание товара при каждом новом запуске парсера;
- хранить историю изменений цен;
- строить аналитику и сравнение по поставщикам в будущем.

## Описание структуры БД

### `supplier_products`

Таблица хранит связь между внутренним материалом и конкретной карточкой товара поставщика.

Основные поля:

- `id`
- `supplier_id` -> FK на `suppliers`
- `material_id` -> FK на `materials`
- `external_name` -> исходное название товара на сайте поставщика
- `product_url` -> уникальный URL товара
- `sku` -> артикул/идентификатор товара у поставщика
- `unit` -> единица измерения
- `attributes_json` -> дополнительные характеристики
- `created_at`
- `updated_at`

### `material_prices`

Таблица хранит историю наблюдаемых цен по товару поставщика.

Основные поля:

- `id`
- `supplier_product_id` -> FK на `supplier_products`
- `price_rub`
- `old_price_rub`
- `price_per_base_unit_rub`
- `stock_status`
- `observed_at`

## Как запустить проект

### 1. Клонировать репозиторий

```bash
git clone <repo_url>
cd "Alex Group project"
```

### 2. Создать и активировать виртуальное окружение

Windows PowerShell:

```powershell
python -m venv venv
.\venv\Scripts\Activate.ps1
```

### 3. Установить зависимости

```bash
pip install -r requirements.txt
```

### 4. Настроить `.env`

Пример переменных:

```env
db_user=postgres
db_password=postgres
db_host=localhost
db_port=5432
db_name=materials_db
```

### 5. Поднять PostgreSQL

Проект ожидает доступную PostgreSQL-базу с указанными параметрами подключения.

### 6. Заполнить стартовые справочники

```powershell
.\venv\Scripts\python.exe app\scripts\seed_data.py
```

Скрипт добавляет:

- материалы;
- поставщиков.

Это важно, потому что импорт Стройландии использует уже существующую запись поставщика и справочник материалов.

### 7. Запустить API

```powershell
.\venv\Scripts\uvicorn.exe app.main:app --reload
```

После старта сервис будет доступен по адресу:

```text
http://127.0.0.1:8000
```

Swagger UI:

```text
http://127.0.0.1:8000/docs
```

## Как запустить парсер через API

Для запуска импорта Стройландии используется endpoint:

```http
POST /parsers/stroylandiya/run
```

Пример запроса:

```bash
curl -X POST http://127.0.0.1:8000/parsers/stroylandiya/run
```

Пример ответа:

```json
{
  "source": "stroylandiya",
  "parsed_items": 128,
  "created_supplier_products": 37,
  "reused_supplier_products": 91,
  "created_material_prices": 128,
  "skipped_items": 6
}
```

## Текущие ограничения

- Production-ready интеграция реализована только для Стройландии.
- Нормализация материалов пока основана на прямом совпадении названий категорий.
- Импорт запускается вручную через API.
- Логирование и фоновое выполнение задач пока не доведены до production-уровня.
- История цен хранится, но аналитические выборки и агрегации ещё не вынесены в отдельный слой.

## Возможные улучшения

- Добавить второго поставщика через единый интерфейс импортера.
- Перевести запуск парсинга в background job или scheduler.
- Добавить retry/backoff и централизованное логирование.
- Ввести нормализацию материалов через mapping layer, а не только по строковому совпадению.
- Добавить парсинг карточек товаров с характеристиками и упаковками.
- Ввести пагинацию и фильтрацию в API списка товаров и цен.
- Перейти на миграции Alembic.
- Добавить тесты на parser, service и repository слой.

## Почему этот проект сильный как portfolio case

Проект показывает не только знание FastAPI и SQLAlchemy, но и умение собрать прикладной backend вокруг реального источника данных:

- работа с внешним HTML-источником;
- асинхронный I/O;
- проектирование таблиц под историю цен;
- разделение на route/service/repository;
- интеграционный сценарий от парсера до API.

Это хороший фундамент для развития в полноценный сервис мониторинга и сравнения цен на строительные материалы по нескольким поставщикам.
